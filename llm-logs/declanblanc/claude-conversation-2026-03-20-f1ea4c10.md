## 🤖 Claude

Based on the code, there are two likely culprits:

### 1. `new AudioContext()` with default output device
When joining a voice channel, the code creates a new `AudioContext()` for speaking detection. On some systems (especially macOS), creating an `AudioContext` can cause the browser to take a more aggressive hold on the audio output device. If the context isn't properly closed on disconnect, it can leave the audio subsystem in a bad state.

### 2. Twilio SDK's audio device management
The bigger suspect is `TwilioVideo.connect()` with `audio: true`. Twilio's SDK acquires the microphone via `getUserMedia` and manages audio I/O internally. Some browsers (particularly Chrome on macOS) change the system audio routing when WebRTC establishes a peer connection — switching the sample rate or audio processing mode at the OS level. This is a **known Chrome/WebRTC issue**: Chrome can switch the macOS CoreAudio HAL to a different sample rate (e.g., 48kHz → 16kHz) to match WebRTC's needs, which disrupts other apps sharing the same audio device.

### Why it persists after leaving
The cleanup in `VoiceContext.tsx` disconnects from the Twilio room, but if the `AudioContext` isn't explicitly `.close()`'d or if Chrome's CoreAudio session doesn't fully release, the audio routing stays in the altered state. That's why switching audio devices "resets" it — it forces the OS to re-initialize the audio pipeline.

### TL;DR
Most likely a **Chrome + WebRTC + macOS CoreAudio interaction** where establishing the peer connection changes the system's audio device configuration, and tearing it down doesn't fully restore it. It's a well-known pain point in the WebRTC ecosystem rather than a straightforward bug in the Harmony code.
